# 迪拜仓库存优先发货系统——项目设计与生成说明

> 本文档用于直接交给 Claude Code / Claude 生成一个可运行项目。  
> 技术栈：**Vue 3 + TypeScript + Vite + Element Plus + FastAPI + Playwright + Pandas + SQLite（可选）**。  
> 运行环境：优先支持 Windows 10/11，项目复制到其他电脑后无需修改绝对路径。

---

## 1. 项目目标

开发一个本地运行的跨境电商发货辅助系统，完成以下流程：

1. 使用 Playwright 打开 Noon Directship 后台。
2. 用户首次手动登录，登录状态保存在项目内部的浏览器配置目录。
3. 程序进入 Pending 订单页面并点击 `Export`。
4. 将导出的 CSV 保存到项目内部目录。
5. 调用 ERP 库存接口获取全部库存明细。
6. 只保留 `warehouseName == "迪拜仓"` 的库存。
7. 按 `bizId` 汇总 `quantity`，生成迪拜仓库存 Excel。
8. 从订单 CSV 中只提取：
   - `order_nr`
   - `partner_sku`
   - `target_shipped_at`
9. 使用以下匹配关系：
   - 订单 `partner_sku`
   - 库存 `bizId`
10. 针对每个 SKU，按 `target_shipped_at` 从早到晚排序。
11. 库存优先分配给最早达到最晚发货期限的订单。
12. 输出“建议发货订单”和“库存不足订单”结果文件。
13. 前端展示运行进度、订单数据、库存数据和最终结果，并允许下载结果文件。

## 2. 当前已知的业务信息

### 2.1 Noon 订单侧

Noon 网站：

```text
https://directship.noon.partners/
```

订单导出入口：

```text
Directship Orders -> Pending -> Export
```

点击 `Export` 后，浏览器网络请求中会出现类似：

```text
export/create
export/status
export/status
...
```

说明 Noon 可能采用异步导出任务：

```text
创建导出任务 -> 轮询状态 -> 文件生成完成 -> 浏览器下载 CSV
```

第一版优先使用 Playwright 模拟浏览器操作并监听下载事件，不直接复现 Noon 私有接口。

订单 CSV 当前只需要以下三个字段：

| 字段 | 含义 | 用途 |
|---|---|---|
| `order_nr` | 订单号 | 标识具体订单 |
| `partner_sku` | 商家 SKU | 与库存 `bizId` 匹配 |
| `target_shipped_at` | 最晚发货时间 | 决定发货优先级 |

当前默认假设：

- CSV 中每一行代表 1 件商品。
- 同一个订单可能出现多行。
- `partner_sku` 与 ERP 库存中的 `bizId` 可以直接匹配。
- `target_shipped_at` 越早，订单优先级越高。

### 2.2 ERP 库存侧

库存接口：

```text
GET http://www.erpzd.com/api/v1/customer/box/skudetail/all
```

鉴权方式中已观察到：

```text
Cookie: token=<TOKEN>
X-Token: <TOKEN>
```

响应大致结构：

```json
{
  "msg": "",
  "code": 20000,
  "data": [
    {
      "data": "[{\"boxCode\":\"...\",\"bizId\":\"GP-Vib405\",\"quantity\":1,\"warehouseName\":\"迪拜仓\"}]"
    }
  ]
}
```

注意：

- 外层响应是 JSON。
- `data` 数组中对象的 `data` 字段可能仍然是 JSON 字符串。
- 程序需要进行第二次 JSON 解析。
- 只保留 `warehouseName == "迪拜仓"`。
- 按 `bizId` 汇总所有 `quantity`。

库存侧核心字段：

| 字段 | 含义 | 用途 |
|---|---|---|
| `bizId` | SKU | 与订单 `partner_sku` 匹配 |
| `quantity` | 当前明细数量 | 汇总为 SKU 总库存 |
| `warehouseName` | 仓库名称 | 只保留迪拜仓 |

第一版暂时不使用 `imgUrl`，避免保存大量长链接。

## 3. 发货分配规则

采用“最早截止时间优先”规则：

```text
Earliest Deadline First
```

### 3.1 基本算法

对每个 `partner_sku`：

1. 获取对应 SKU 的迪拜仓库存数量。
2. 找出所有相同 `partner_sku` 的待发货订单。
3. 按 `target_shipped_at` 升序排序。
4. 若时间相同，再按 `order_nr` 升序排序，保证结果稳定。
5. 从最早的订单开始依次占用库存。
6. 库存用完后，剩余订单标记为库存不足。

示例：

```text
SKU: GP-Vib304
迪拜仓库存: 3
待发货订单: 4
```

订单时间：

```text
A001 -> 2026-06-20 10:00
A002 -> 2026-06-21 10:00
A003 -> 2026-06-22 10:00
A004 -> 2026-06-24 10:00
```

结果：

```text
建议发货: A001、A002、A003
库存不足: A004
```

### 3.2 数据清洗规则

订单侧：

- 去除 `order_nr` 为空的数据。
- 去除 `partner_sku` 为空的数据。
- 去除 `target_shipped_at` 无法解析的数据。
- `partner_sku` 前后空格需要删除。
- 默认不强制转换 SKU 大小写，以免破坏真实编码。
- 解析时间时保留时区信息。
- 若时间没有时区，按照配置项 `ORDER_TIMEZONE` 处理。

库存侧：

- 只保留迪拜仓。
- `quantity` 为空时按 0 处理。
- `quantity` 非数字时记录错误并跳过。
- `bizId` 前后空格需要删除。
- 同一 `bizId` 的数量求和。
- 负数库存不得参与分配，按 0 处理并记录警告。

### 3.3 输出状态

结果至少包含：

```text
SELECTED
SHORTAGE
INVALID
```

含义：

- `SELECTED`：库存足够，建议发货。
- `SHORTAGE`：库存不足，暂不建议发货。
- `INVALID`：缺少关键字段或时间解析失败。

## 7. Noon CSV 下载模块

文件：

```text
backend/app/services/noon_downloader.py
```

### 7.1 工作方式

使用 Playwright 同步 API或异步 API均可，但必须避免阻塞 FastAPI 主线程。

推荐方式：

- FastAPI 创建后台任务。
- 后台任务在线程中运行 Playwright。
- 前端通过任务 ID 轮询进度。

### 7.2 浏览器配置

使用持久化上下文：

```text
runtime/noon_profile/
```

优先使用 Microsoft Edge：

```text
channel = "msedge"
```

配置：

```text
headless = false
accept_downloads = true
```

首次运行：

1. 打开 Edge。
2. 用户手动登录 Noon。
3. 用户进入 Pending 页面。
4. 前端提示用户点击“已完成登录，继续执行”。
5. 程序寻找 `Export` 按钮并点击。

后续运行：

- 读取 `runtime/noon_profile` 中的登录状态。
- 优先直接访问 `NOON_PENDING_URL`。
- 若发现登录失效，任务状态改为 `WAITING_LOGIN`，提示用户重新登录。

### 7.3 下载处理

必须先监听下载事件，再点击按钮：

```text
监听 download -> 点击 Export -> 等待下载完成 -> save_as
```

保存到：

```text
data/orders/current/noon_pending_orders.csv
```

若服务器返回的真实文件是 `.xlsx`，不得强行改名为 `.csv`，应根据原文件后缀保存，并由解析模块自动识别。

### 7.4 异常处理

出现以下情况时：

- 页面加载失败。
- 登录失效。
- 找不到 Export 按钮。
- 点击后 300 秒内没有下载。
- Noon 显示导出失败。

必须：

1. 设置明确错误信息。
2. 保存当前页面截图到：
   ```text
   runtime/screenshots/
   ```
3. 前端显示可读错误。
4. 不覆盖上一次成功下载的 current 文件。

## 8. ERP 库存获取模块

文件：

```text
backend/app/services/inventory_client.py
```

请求：

```text
GET {ERP_BASE_URL}{ERP_INVENTORY_ENDPOINT}
```

请求头至少包含：

```text
Accept: application/json, text/plain, */*
X-Token: <ERP_TOKEN>
User-Agent: Mozilla/5.0
Referer: http://www.erpzd.com/
```

Cookie：

```text
token=<ERP_TOKEN>
sidebarStatus=0
```

处理要求：

1. 检查 HTTP 状态码。
2. 检查业务状态码是否为 `20000`。
3. 遍历外层 `data`。
4. 若内部 `data` 是字符串，再执行 `json.loads`。
5. 合并所有内部列表。
6. 过滤 `warehouseName == "迪拜仓"`。
7. 按 `bizId` 汇总 `quantity`。
8. 生成处理后的 Excel。

输出：

```text
data/inventory/current/dubai_inventory.xlsx
```

Excel 只需要：

| bizId | quantity |
|---|---:|

若 `KEEP_RAW_FILES=true`，同时保存原始响应：

```text
data/inventory/archive/inventory_raw_时间戳.json
```

- `target_shipped_at` 转为统一可排序时间。
